# PLC代码审查工具

自动化检查PLC XML源码与Excel IO表、IO数据库的匹配性，识别命名规范问题。

## Thinking Discipline
- 每次 tool call 返回后，thinking 阶段限制在 500 tokens 以内
- 严禁在 thinking 中重复任何句子超过 2 次
- tool call 成功后，直接执行下一个工具，不需要在 thinking 中反复确认

## Quick Start

### 1. 准备文件

**PLC源码文件**: XML格式（从PLC导出）

**Excel文件**（重要：两个文件用途不同）：

| 文件类型 | 示例文件名 | 用途 | 关键列 |
|---------|-----------|------|--------|
| **IO表** | `1098 IO表-1212.xlsx` | 用于比对**IO名称**和**IO地址** | 参考 **IO表Excel列识别逻辑** 章节列定义 |
| **IO数据库** | `IO数据库V1.0.03.xlsx` | 用于比对**IO名称**和**PLC控制变量** | CM列（注释）、PLC变量列、HMI地址列 |

**⚠️ 重要区别**：
- **IO表**：简单表格，IO地址与IO名称一一对应（如 OD10109 = 机架外罩_贴胶集成自动备胶按钮灯）
- **IO数据库**：复杂结构，包含PLC变量名、HMI地址、CM注释等多个字段，用于验证PLC代码中的变量名是否与数据库一致

**使用方式**（支持多个PLC XML文件）：
```bash
# 单个PLC文件 + 两个Excel文件（推荐）
python scripts/plc_audit.py code.xml "1098 IO表-1212.xlsx" "IO数据库V1.0.03.xlsx"

# 多个PLC文件 + 两个Excel文件
python scripts/plc_audit.py code1.xml code2.xml "1098 IO表-1212.xlsx" "IO数据库V1.0.03.xlsx"

# 只使用IO表（仅比对IO名称和地址）
python scripts/plc_audit.py code.xml "1098 IO表-1212.xlsx"

# 只使用IO数据库（仅比对PLC控制变量）
python scripts/plc_audit.py code.xml "IO数据库V1.0.03.xlsx"
```

**⚠️ 文件名含中文或特殊字符**：
Excel 文件名含中文、空格、括号等时，**必须用双引号包裹**，否则 Shell 会解析失败：
```bash
# ✅ 正确
python scripts/plc_audit.py code.xml "国轩-IO表-A.xlsx" "IO数据库V1.0.03.xlsx"

# ❌ 错误（中文被截断/乱码，导致"Excel文件读取失败"）
python scripts/plc_audit.py code.xml 国轩-IO表-A.xlsx IO数据库V1.0.03.xlsx
```
**原则：所有非纯英文文件名的参数一律加双引号。**

### 2. 运行审查

**只审查指定类型变量（推荐）：**
```bash
# 单个PLC文件 + 单个Excel文件
python scripts/plc_audit.py <plc_xml_file> <io_table.xlsx>

# 多个PLC文件 + 多个Excel文件（IO表 + IO数据库）
python scripts/plc_audit.py <plc_xml_file1> [plc_xml_file2 ...] <io_table.xlsx> <io_database.xlsx>

# 指定输出目录
python scripts/plc_audit.py code.xml io_table.xlsx io_db.xlsx -o ./reports

# 多个PLC文件指定输出目录
python scripts/plc_audit.py code1.xml code2.xml io_table.xlsx io_db.xlsx -o ./reports
```

**注意**：文件名含中文/空格/括号时，必须加双引号，参见上文【文件名含中文或特殊字符】章节。

**完整审查所有Sheet：**
```bash
python scripts/plc_audit_full.py <plc_xml_file> <excel_file1> [excel_file2 ...] [-o output_dir]
```

**仅提取变量（不比对）：**
```bash
python scripts/plc_extract.py <plc_xml_file> [-o output_dir]
```

### 3. 查看报告
审查完成后会生成：
- `plc_audit_report_YYYYMMDD_HHMMSS.md` - 审查报告
- `plc_source_cleaned_YYYYMMDD_HHMMSS.txt` - 清理后的PLC源码

## 架构原则

> **重要**: 源码解析必须保证完整性
> 1. **第一阶段** (`parse_plc_variables`): 完整解析所有内容，不做任何过滤
> 2. **清理文件** (`clean_plc_source`): 保留所有源码行（包括被注释的代码），只过滤纯注释行和空行
> 3. **后续分析**: 如需过滤，必须复制数据进行处理，不得修改原始解析结果
> 4. **目的**: 便于回归测试，确保源码结构完整可追溯

## PLC代码解析规范

### IO地址格式
以下格式**一定是IO地址**，不应被识别为变量名：
- `IAxxx` - 模拟输入地址（如 `IA2301`, `IA4212`）
- `IDxxx` - 数字输入地址
- `ODxxx` - 数字输出地址（如 `OD10605`）
- `OFxxx` - 模拟输出地址

### 变量名格式
变量名由**模块前缀** + **功能名** + **可选后缀**组成：
- **模块前缀**: `ZFJ_`, `YFJ_`, `TJ_`, `JR_`, `XX_`, `QJ_`, `QD_` 等
- **功能名**: 描述设备功能，如 `YHJQZ` (压辊气缸), `ZYXAN` (正压按钮)
- **可选后缀**（结构体成员）:
  - `.Sr_I` - 气缸回限（输入）
  - `.Sr_O` - 气缸出限（输入）
  - `.OUT` - 输出信号
  - `.Bn_On` - 按钮开启状态
  - `.In` - 输入
  - 也可能**无后缀**：如 `MB_ix_WXAN := IA2208`

### 数组变量处理
数组变量（如 `SFJ_KDJC[0]`）**不是IO变量**，不应与Excel IO表比对：
- 格式: `变量名[索引]` 或 `变量名[索引1,索引2]`
- 示例: `IA_Temp[0,0]`, `SFJ_KDJC[0]`, `g_arAD[1]`
- **处理规则**: 参与命名规范检查，但不参与IO地址匹配

### 行尾注释提取规则
工具解析 `//` 后的行尾注释，用于与Excel IO名称比对：

```pascal
// 有行尾注释的情况
YTJ_QJ_SJ1.Sr_I := IA0000; //拐角贴胶缓存_拐角胶上切胶气缸回限
                          ↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑↑
                          提取为 "拐角贴胶缓存_拐角胶上切胶气缸回限"

// 无行尾注释的情况
YTJ_QJ_SJ1.Sr_O := IA0001;
                          (无注释，显示为 "-")
```

**关键规则**：
1. **变量定义**：行尾没有 `//` 的代码行视为变量定义，注释留空（显示为 `-`）
2. **IO地址行**：行尾有 `//` 的代码行，提取 `//` 后的内容作为注释
3. **比对依据**：用提取的注释与Excel IO名称进行模糊匹配

## IO表Excel列识别逻辑

工具自动识别Excel中的IO列、变量名列、注释列和反气缸列，并且**只处理指定的Sheet类型**。

### 指定处理的Sheet类型

工具将只处理以下四个特定Sheet类型：
1. **气缸Sheet** - 包含气缸相关的IO地址和变量
2. **真空吹气Sheet** - 包含真空吹气相关的IO地址和变量  
3. **数字量Sheet** - 包含数字量IO地址和变量
4. **模拟量Sheet** - 包含模拟量IO地址和变量

### 气缸Sheet列识别规则

| 列类型 | 列位置 | 识别关键词 | 示例 |
|--------|--------|------------|------|

| **电磁阀名称列** | B列 | `电磁阀名称`, `名称` | "电磁阀名称", "名称" |
| **变量列1** | C列 | `变量`, `VAR`, `PLC` | "变量", "PLC变量" |
| **变量列2** | D列 | `变量`, `VAR`, `PLC` | "变量", "PLC变量" |
| **出限列** | E列 | `出限`, `OUT`, `伸出` | "出限", "OUT" |
| **出限名称列** | F列 | `出限名称`, `名称` | "出限名称", "名称" |
| **回限列** | G列 | `回限`, `IN`, `缩回` | "回限", "IN" |

| **名称列2** | D列 | `名称`, `变量名` | "名称", "变量名" |
| **变量列2** | E列 | `变量`, `VAR`, `PLC` | "变量", "PLC变量" |

**⚠️ 重要说明**：在IO表（第一个Excel文件）中，真空吹气Sheet的**"名称"列是IO名称/注释列**，不是变量名列。这与IO数据库（第二个Excel文件）的结构不同。

### 数字量Sheet列识别规则

| **IO地址列** | A列 | `IO`, `地址`, `ADDR` | "IO地址", "地址" |
| **名称列（注释）** | B列 | `名称` | **"名称"列在IO表中被识别为注释列，不是变量名列** |

**⚠️ 重要说明**：在IO表（第一个Excel文件）中，数字量Sheet的**"名称"列是IO名称/注释列**，不是变量名列。

| 列类型 | 列位置 | 识别关键词 | 示例 |
|--------|--------|------------|------|
| **模块列** | A列 | `模块`, `名称` | "模块", "名称" |
| **名称列（注释）** | B列 | `名称` | **"名称"列在IO表中被识别为注释列，不是变量名列** |
| **变量列** | C列 | `变量`, `VAR`, `PLC` | "变量", "PLC变量" |

**⚠️ 重要说明**：在IO表（第一个Excel文件）中，模拟量Sheet的**"名称"列是IO名称/注释列**，不是变量名列。

## IO数据库Excel列识别规则（第二个文件）

IO数据库文件（如`IO数据库V1.0.03.xlsx`）的列结构与IO表**完全不同**：

| 列名 | 用途 | 示例值 |
|------|------|--------|
| **ID** | 记录ID | 1, 2, 3... |
| **设备名称** | 设备中文名称 | "拐角贴胶缓存_拐角胶上切胶气缸回限" |
| **CM** | IO名称/注释（关键列） | "拐角贴胶缓存_拐角胶上切胶气缸回限" |
| **PLC控制变量** | PLC源码变量名（关键列） | `YTJ_QJ_SJ1.Sr_I` |
| **HMI控制变量** | HMI变量名 | `YTJ_QJ_SJ1.Sr_I` |
| **HMI强制输出** | HMI输出地址 | `IA0000` |

**列识别策略**：
- **IO地址识别**：从 `HMI强制输出` 列或 `PLC控制变量` 列中提取
- **注释识别**：`CM` 列（主要）、`设备名称` 列（备用）
- **PLC变量识别**：包含"PLC控制变量"的列（如 `气缸PLC控制变量`, `真空吹气PLC控制变量` 等）

**前缀列名支持**：
工具通过**子字符串匹配**识别带前缀的列名：
- `气缸PLC控制变量` ✅ 匹配（包含'PLC控制变量'）
- `真空吹气PLC控制变量` ✅ 匹配（包含'PLC控制变量'）
- `模拟量PLC控制变量` ✅ 匹配（包含'PLC控制变量'）
- `PLC控制变量` ✅ 匹配（包含'PLC控制变量'）

### Sheet类型差异

IO数据库不同Sheet的列结构略有不同：

| Sheet  | 注释列  | PLC变量列 |
|--------|---------|----------|
| 气缸   | CM | 气缸PLC控制变量 |
| 真空吹气  | CM | 真空吹气PLC控制变量 |
| 数字量  | CM | PLC控制变量 |
| 模拟量  | CM | 模拟量PLC控制变量 |

**注意**：IO数据库的解析逻辑与IO表**完全不同**，工具内部通过 `file_type` 标记区分处理。

### IO数据库特殊列结构说明（关键！）

**IO数据库与IO表的核心区别**：

| 特性 | IO表 | IO数据库 |
|------|------|----------|
| IO地址列 | ✅ 有（如IA/ID/OD列） | ❌ **没有** |
| IO名称列 | 名称/注释列 | **CM列** |
| IO变量列 | 变量列 | **PLC控制变量列** |
| 用途 | 比对IO地址和名称 | 比对IO名称和PLC变量 |

**气缸/真空吹气/模拟量Sheet的特殊处理**：

这些Sheet**没有IO地址列**，只有：
- **CM列**：IO名称/注释（如"拐角贴胶缓存_拐角胶上切胶气缸回限"）
- **PLC控制变量列**：PLC变量名（如`YTJ_QJ_SJ1.Sr_I`）

工具处理方式：
1. 识别到 `CM` 列作为注释列
2. 识别到包含"PLC控制变量"的列作为IO变量列
3. **不强制要求IO地址格式**：PLC控制变量列的值即使不是IA/ID/OD格式（如`YTJ_QJ_SJ1.Sr_I`），也会被正确解析为变量名
4. 记录中 `io_address` 字段可能为空或显示变量名（因为无真实IO地址）

**数字量Sheet的特殊性**：

数字量Sheet**有IO地址**（在PLC控制变量列中），格式如 `IA0000`、`OD10109` 等，因此可以正常提取IO地址。

### IO数据库列名识别策略

工具使用**多策略级联匹配**识别IO数据库列：

1. **策略1（精确匹配）**：'CM' == 列名
2. **策略2（关键词匹配）**：'PLC控制变量' in 列名
3. **策略3（前缀匹配）**：列名以Sheet名开头 + 'PLC控制变量'
4. **策略5.5（模糊匹配）**：同时包含'PLC'、'控制'、'变量'
5. **策略6（配对匹配）**：IO列与CM列配对识别

**关键修复**：工具优先识别IO数据库的`CM`列作为注释列，而非IO表的变量列，确保正确解析。

## 数据结构说明

### IO表 vs IO数据库数据分离

**设计原则**：
- IO表和IO数据库的数据**必须分开存储**，避免互相覆盖
- IO表数据主要用于**IO地址匹配**
- IO数据库数据主要用于**PLC控制变量匹配**

**返回的数据结构** (`parse_excel_io_db` 函数返回)：

```python
{
    # 合并数据（向后兼容）
    'by_address': {...},              # 所有IO地址（去重）
    'by_name': {...},                 # 所有变量名（去重）
    'by_name_all': {...},             # 所有变量名（保留重复）
    'by_address_all': {...},          # 所有IO地址（保留重复）
    
    # IO表专用数据（用于IO地址匹配）
    'by_address_table_only': {...},  # IO表IO地址
    'by_name_table_only': {...},      # IO表变量名
    
    # 【关键】IO数据库专用数据（用于PLC控制变量匹配）
    'by_name_db_only': {...},         # IO数据库变量名（保留重复）
    'by_address_db_only': {...},       # IO数据库IO地址（保留重复）
    
    'sheet_analysis': [...],          # Sheet分析详情
    'var_name_duplicates': {...},     # 变量名重复记录
    'stats': {...},                   # 统计信息
}
```

**关键字段说明**：

| 字段 | 用途 | 数据来源 |
|------|------|----------|
| `by_name_db_only` | PLC控制变量匹配 | IO数据库文件 |
| `by_name_table_only` | IO地址映射 | IO表文件 |
| `by_name_all` | 综合查询 | IO表 + IO数据库（合并） |

**source_type 标记**：
- 每个记录都有 `source_type` 字段：`'io_table'` 或 `'io_database'`
- 用于区分数据来源，支持筛选和优先级排序

### 数据分离逻辑

**解析时分离** (`parse_excel_io_db`)：
```python
if file_type == 'io_database':
    # 保存到 IO 数据库专用变量
    io_db_by_name_all[cleaned] = record
    io_db_by_address_all[io_addr] = record
else:
    # IO 表数据
    io_table_by_name_all[cleaned] = record
    io_table_by_address_all[io_addr] = record
```

**使用时分离** (`generate_report`)：
```python
source_io_mismatches = check_source_io_match(
    plc_vars, 
    io_by_address, 
    io_by_name, 
    ...,
    io_db.get('by_name_db_only')  # 使用 IO 数据库专用数据
)
```

## 重复项处理

工具支持检测和处理以下类型的重复：

1. **IO地址重复**：同一个IO地址在多个行中出现
2. **IO名称重复**：同一个IO名称（注释）在多个行中出现  
3. **PLC变量重复**：同一个PLC控制变量在多个行中出现

**重复处理策略**：
- **优先级排序**：IO地址匹配 > 注释匹配 > 有PLC控制变量 > 注释详细程度 > Sheet优先级 > 行号
- **报告展示**：在第10节"数据质量问题 - 重复项"中列出所有重复
- **匹配选择**：优先选择IO数据库的记录（`source_type='io_database'`）

## 报告生成

### 报告结构

1. **统计摘要**：总体统计信息
2. **IO接线问题**：Sr_O/Sr_I接反、反气缸NOT、正常气缸NOT
3. **仅在Excel中的地址**：Excel有但PLC未使用的IO
4. **仅在PLC中的地址**：PLC使用但Excel未定义的IO
5. **源码IO及PLC控制变量核对**：详细匹配结果
6. **变量名与设备描述不匹配**：命名规范问题
7. **IO数据库中有但PLC中未找到的变量**：数据库中有PLC源码中无
8. **命名规范问题**：模块前缀缺失等
9. **审核标准**：审核规则说明
10. **数据质量问题 - 重复项**：重复IO地址、名称、变量

### 第5节"源码IO及PLC控制变量核对"说明

本节显示PLC源码变量与Excel数据的详细比对结果。

**列说明**：
| 列 | 说明 |
|----|------|
| 源码变量 | PLC源码中的变量名，如 `YTJ_QJ_SJ1.Sr_I` |
| IO名称 | 从源码行尾注释提取的IO名称 |
| IO地址 | 源码中赋值的IO地址 |
| IO地址是否匹配 | IO地址在Excel中是否找到 |
| IO名称是否匹配 | IO名称在Excel中是否匹配 |
| 变量是否匹配 | PLC控制变量是否匹配 |
| IO地址备注 | IO地址匹配详情（行号等） |
| IO名称备注 | IO名称匹配详情（行号等） |

**变量匹配逻辑**：
1. 优先使用 IO 数据库数据（`by_name_db_only`）
2. 比较 `plc_control_var` 字段与源码变量名
3. 支持完整变量名匹配（如 `YTJ_QJ_SJ1.Sr_I`）
4. 显示正确的行号（如"数字量第737行"）

**第5节记录逻辑（问题数据定义）**：

第5节捕获需要人工核对的"问题数据"，包括以下类型：

**IO表相关问题**：
| 问题类型 | 判断条件 | 显示示例 |
|---------|---------|---------|
| IO名称匹配到多个 | IO表中找到多个相同IO名称 | IO地址匹配到多行:气缸第68行, 数字量第2行 |
| IO名称未找到 | IO表中找不到该IO名称 | IO名称未找到 |
| IO地址不一致 | IO数据库匹配但IO地址与源码不同 | IO地址备注显示实际地址 |

**IO数据库相关问题**：
| 问题类型 | 判断条件 | 显示示例 |
|---------|---------|---------|
| 变量不匹配 | IO数据库中变量名与源码不同 | 变量不匹配(数据库:YTJ_QJ_SJ1.Sr_O) |
| 变量为空 | IO数据库中PLC控制变量为空 | 变量不匹配(数据库:变量为空) |
| IO名称匹配到多个 | IO数据库中找到多个相同IO名称 | 数据库位置:匹配到2个变量:数字量第737行等 |
| IO名称未找到 | IO数据库中找不到该IO名称 | IO名称未找到 |

**完全匹配但显示**：
即使IO数据库完全匹配，如果IO表匹配到多行，仍然显示在第5节（让用户知道数据来源）：
```
| YTJ_QJ_SJ1.Sr_I | 拐角贴胶缓存_拐角胶上切胶气缸回限 | IA0000 | 是 | 是 | 匹配(数字量:737行) | IO地址匹配到多行:气缸第68行, 数字量第2行 | 数据库位置:匹配到1个变量:数字量第737行 |
```

**注意**：
- "是" = 匹配正确，"否" = 不匹配，"部分匹配" = 部分正确
- IO地址备注优先显示IO数据库位置，其次显示IO表位置
- IO名称备注显示变量匹配的具体位置（行号）

## 故障排除

### 问题：IO数据库解析失败，"cannot access local variable 'plc_control_var_col'"

**症状**：
- IO数据库数字量 sheet 解析出错：`cannot access local variable 'plc_control_var_col'`
- `by_name_db_only` 为空或记录数很少
- 第5节显示"变量不匹配(数据库:变量为空)"

**原因**：
`plc_control_var_col` 在列配对循环内部定义，但在行解析循环外部使用时无法访问（作用域问题）。

**修复方法**：
1. 将 `plc_control_var_col` 的定义移到列配对循环外部（`new_io_cols = []` 之后）
2. 在行解析前初始化：`plc_control_var_col = None`
3. 使用 `try/except` 捕获未定义错误（备用方案）

**代码位置**：`parse_excel_io_db` 函数，列配对循环和行解析循环之间。

### 问题：第5节显示"变量不匹配(数据库:变量为空)"

**症状**：
- IO数据库中有PLC控制变量，但报告显示"变量为空"
- 行号显示不正确（如"数字量第2行"而不是"数字量第737行"）

**原因**：
1. `plc_control_var` 字段未正确提取（见上一条）
2. `db_records` 使用了合并数据而非IO数据库专用数据

**修复方法**：
1. 修复 `plc_control_var_col` 作用域问题（确保变量正确提取）
2. 在 `generate_report` 中传递 `io_db.get('by_name_db_only')` 而非 `io_by_name_all`
3. 确保 `check_source_io_match` 使用IO数据库专用数据进行变量匹配

### 问题：IO表和IO数据库数据互相覆盖

**症状**：
- IO数据库记录被IO表记录覆盖
- 变量名重复时，IO表数据优先于IO数据库数据
- 统计信息不正确

**原因**：
IO表和IO数据库数据存储在同一个字典中，后处理的文件会覆盖先处理的文件。

**修复方法**：
1. 在 `parse_excel_io_db` 中分离存储：
   - `io_table_by_name_all` / `io_table_by_address_all`：IO表数据
   - `io_db_by_name_all` / `io_db_by_address_all`：IO数据库数据
2. 根据 `file_type` 分别保存到对应变量
3. 返回时包含分离的数据结构：
   ```python
   {
       'by_name_db_only': io_db_by_name_all,      # IO数据库专用
       'by_address_db_only': io_db_by_address_all,   # IO数据库专用
       'by_name_table_only': io_table_by_name_all,  # IO表专用
       'by_address_table_only': io_table_by_address_all,  # IO表专用
   }
   ```
4. 在 `generate_report` 和 `check_source_io_match` 中使用正确的数据源

### 问题：列识别失败，"未识别到IO列"

**症状**：
- 气缸、真空吹气、模拟量 sheet 显示"未识别到IO列，跳过"
- IO数据库只有数字量 sheet 被解析

**原因**：
IO数据库的列名包含前缀（如"气缸PLC控制变量"、"真空吹气PLC控制变量"），工具的策略5.5匹配逻辑在某些情况下失效。

**修复方法**：
1. 提前识别 `plc_control_var_col`（包含"PLC控制变量"的列）
2. 确保列识别在所有sheet中都能工作
3. 使用子字符串匹配而非精确匹配：`if 'PLC控制变量' in col_str`

### 问题：第7节"IO数据库中有但PLC中未找到的变量"为空

**症状**：
- 第7节没有显示任何记录
- IO数据库中有变量，但报告说"未找到"

**原因**：
IO数据库列识别失败，导致 `by_name_db_only` 为空或数据不正确。

**修复方法**：
1. 先修复IO数据库解析问题（见上文）
2. 确保 `plc_control_var` 字段正确填充
3. 检查 `generate_report` 中第7节的逻辑是否正确使用 `by_name_db_only`
